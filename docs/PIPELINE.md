## 0. TL;DR（30 秒读完版）

用户在前端输入一条关于企业 ESG 的自然语言问题，后端依次走：**问题改写 → 意图路由 → 多策略检索（向量 + 关键词混合）→ 图谱补充 → 大模型生成 → 引用回填**。系统会基于会话历史维护多轮上下文；遇到无法回答的问题会被记录用于后续人工补答与知识库回流（HITL 闭环正在开发中，见 §4 与 Roadmap）。

整条链路在 P50 约 3 秒、P95 约 5 秒返回；离线侧有独立的文档入库管道，新文档入库后 [N] 分钟内即可被检索到。

---

## 1. 系统定位

| 项 | 内容 |
|---|---|
| 输入 | 自然语言提问（中文 / 英文），可带文档范围筛选 |
| 输出 | 文本答案 + 引用证据块 + 图谱关系 + 元信息（策略、降级路径、耗时）|
| 适用场景 | 企业 ESG 报告问答、研究文档查询、培训资料检索、跨企业指标对比、趋势预测 |
| 不适用场景 | 实时新闻类、需要联网的问题、非 ESG 领域的开放问答（会走闲聊路由） |

## 3. 数据流：两条独立的 Pipeline

### 3.1 离线 · 文档入库管道

| 步骤 | 模块 | 输入 | 输出 | 说明 |
|---|---|---|---|---|
| 1 | 上传 | PDF / Markdown / 网页 | 原始文件 | 通过 Admin 后台或自动同步 |
| 2 | 解析 | `ai_service/extractor.py` | 纯文本 + 结构信息 | 按章节、段落、表格保留层级 |
| 3 | 清洗 | 同上 | 规整文本 | 去页眉页脚、合并断行、统一编码 |
| 4 | 分块 | 分块器 | 语义块列表 | 通常按段落 + 字符阈值；表格独立成块 |
| 5 | 元数据标注 | 自动 + 规则 | 元数据字段 | 企业名、年份、E/S/G 维度、章节来源 |
| 6 | 向量化 | bge-m3（本地） | 向量 | 批量入向量库 |
| 7 | BM25 索引 | `rag/bm25_index.py` | pickle 索引文件 | 持久化到磁盘 |
| 8 | 图谱抽取 | LLM + 规则 | 三元组（企业–指标–关系）| 写入 Neo4j |
| 9 | 文档登记 | `document_registry.py` | 注册条目 | 便于后续按文档 ID 过滤 |

**关键性质：**
- 所有步骤幂等可重跑
- 入库失败的文档进重试队列，不阻塞其他文档
- 新文档入库到可被检索通常 < [N] 分钟

### 3.2 在线 · 问答主链路

下面按"一条 query 进来发生了什么"线性讲。

#### 阶段 A：会话准备

| 步骤 | 模块 | 行为 |
|---|---|---|
| A1 | 会话存储（Redis）| 按 `session_id` 拉取历史消息 |
| A2 | 上下文压缩 | 短期对话用滑动窗口（最近 N 轮）；更早历史的摘要压缩在 Roadmap 中 |
| A3 | 组装 `history_block` | 给后续 rewrite / route / 生成使用 |

> **平均输入 token 较朴素拼接历史减少约 35%**；长对话场景显著降低延迟与成本。

#### 阶段 B：问题改写（Rewrite）

输入：原始 query + history_block
输出：消除指代、补全省略后的"检索可用 query"

举例：
- 原始："那它的范围一排放呢？"
- 改写："BHP 2024 年的范围一（Scope 1）温室气体排放量是多少？"

实现：调用大模型，输出仅一行改写结果；失败时回退到原 query。

> **优化点**：当 query 较短且无指代且无历史时，直接跳过 rewrite（已规划，见 Roadmap）。

#### 阶段 C：意图路由（Router）

Router 决定本次请求走哪条检索策略。先用规则判别，规则判不出再调用 LLM 分类器，最后规则兜底。

**规则信号（部分）：**

| 信号 | 触发策略 |
|---|---|
| 闲聊 / 元问题（"你是谁"、"谢谢"）| `no_retrieval` → 直接走闲聊回复 |
| 复合句（含"对比"、"分别"、"以及"）| `decomposition` |
| 短问题或含代词 | `multi_query`（扩展为多个变体后并发检索）|
| 含年份、数字、缩写（GHG / TCFD / SBTi）| `hybrid`（关键词敏感）|
| 指定文档或带强实体提示 | `graph_first` |
| 预测模式 | `layered`（分主线 / 学术先验 / 法规三层）|
| 其他 | `vector_only` |

**每条策略带 fallback 链**，例如：
- `layered` 失败 → `multi_query` → `hybrid` → `vector_only`
- 任何一层只要返回了非空结果就停止下探

#### 阶段 D：策略执行

不同策略的具体行为：

| 策略 | 行为 |
|---|---|
| `vector_only` | 单次向量召回，取 top-k，去重返回 |
| `hybrid` | 并行跑向量召回 + BM25，RRF 融合（默认）或加权融合 |
| `multi_query` | LLM 生成 N 个 query 变体 → 各自走 hybrid 检索 → RRF 融合 |
| `decomposition` | LLM 把问题拆成子问题 → 每个子问题再走 multi_query |
| `graph_first` | 优先用图谱实体匹配限定文档范围，再做 hybrid |
| `layered` | 同时检索 ESG 主线、学术先验、法规三层（用于趋势预测）|
| `no_retrieval` | 不检索，直接走闲聊模板 |

> **Recall\@5 从约 70% 提升到 88%**，主要来自 hybrid + 重排 + 多查询扩展的组合。

#### 阶段 E：图谱补充（Graph Context）

并行（或紧接）地从 Neo4j 拉一个相关子图：

1. 对 query 做实体识别（公司、指标、年份）
2. 在图里查这些实体，按跳数（默认 1–2 跳）扩展邻居
3. 按"端点是否命中匹配实体 + 边置信度"排序
4. 取前若干条边，格式化成可读三元组（`(BHP) --reports--> (Scope 1: 12.3 Mt CO2e, 2024)`），喂给生成阶段

如果 query 没匹配到任何实体，跳过。

#### 阶段 F：答案生成（多级降级链）

按优先级尝试：

| 级别 | 模型 | 适用场景 |
|---|---|---|
| 1 | 云端大模型（OpenAI 兼容接口）| 主路径，质量最佳 |
| 2 | 本地 QLoRA 模型 | 云端限流 / 网络异常 / 成本敏感场景 |
| 3 | 关键词抽取兜底 | 上述全挂时，从 sources 抽最相关句子拼接 |

**特殊模式：**
- `ask` 模式：标准问答，输出文本 + 引用
- `predict` 模式：基于 layered 检索 + 图谱因果链做结构化预测，输出包含数值预测、置信度、依据链路的 JSON

**Prompt 关键约束：**
- 严格要求"只用提供的 excerpts 与图谱"
- 没证据时输出固定文案 `"The provided reports do not contain enough information to answer this question."`
- 可选 `RAG_ALLOW_SPECULATION` 开关，开启时允许大模型标注"Tentative hypothesis"，关闭时禁止

#### 阶段 G：响应组装

返回给前端的 JSON 包含：

```jsonc
{
  "answer": "...",
  "sources": [{ "chunk_id", "text", "document_id", "document_title", "fusion_score" }],
  "graph_sources": { "matched_entities": [...], "edges": [...] },
  "backend": "openai+graph",        // 实际走到的生成级别
  "mode": "ask",
  "original_query": "...",
  "rewritten_query": "...",
  "sub_queries": ["...", "..."],
  "retrieval_strategy": "layered",
  "fusion_method": "rrf_hybrid",
  "routing": { "strategy", "reason", "fallback_chain", "fallbacks_used" },
  "timings_ms": { "rewrite": 412, "route": 8, "retrieval": 1820, "graph": 380, "generate": 1510, "total": 4140 }
}
```

`timings_ms` 是排障神器，所有问题报到群里时请附上。

## 4. Human-in-the-loop 兜底闭环（开发中 / Work In Progress）

> ⚠️ 本节描述的是目标设计，当前仅 4.1 触发信号的检测已落地（见 backend 字段），4.2 起的入队 / 去重 / 邮件 / 回流尚未实现，正按 TASKS_FOR_CODEX.md Task 2 推进。

### 4.1 触发条件

主链路一旦命中以下任一信号，向 MCP Notifications 服务上报：

| 信号 | 含义 |
|---|---|
| `no_context` | 检索后没有任何 sources |
| `INSUFFICIENT_CONTEXT_ANSWER` | LLM 主动声明证据不足 |
| `predict_parse_failed` | 预测模式结构化输出解析失败 |
| `extractive_fallback` | 全部生成链路降级到关键词抽取 |
| 低质量（可选）| sources 数 ≤ 1 且融合分数低于阈值 |

### 4.2 处理流程

1. **入队（即时）**：上报调用 fire-and-forget，不阻塞用户响应
2. **去重（小时级）**：对 query 做指纹哈希，同指纹合并，累加出现次数与提问用户
3. **批量（默认每 15 分钟）**：worker 拉取 pending 条目，按"被多少个不同用户问过"排序，组装 digest 邮件
4. **通知（按域路由）**：根据问题所属 ESG 维度 / 文档域路由到对应研究员邮箱
5. **人工补答**：研究员在 Admin 后台查看 → 标注答案 / 上传新文档 / 标记 wontfix
6. **审核 + 回流**：补充内容审核后，触发离线入库管道，新文档 [N] 分钟后即可被检索

### 4.3 监控指标

| 指标 | 目标 |
|---|---|
| 入队成功率 | > 99.5% |
| 去重压缩比 | > 80% |
| 入队到邮件中位数 | < 15 分钟 |
| 平均解决时延（人工补答）| < 3 工作日 |
| 同指纹解决后复发率 | < 20% |
| 月均由通知驱动的文档新增数 | [N] |

## 5. 关键设计决策与权衡

| 决策点 | 选择 | 替代方案 | 为什么这么选 |
|---|---|---|---|
| 检索融合方法 | RRF（默认）| 加权归一化 | RRF 不需要分数归一化，对异构 retriever 鲁棒；加权方式作为开关保留 |
| Embedding | 本地 bge-m3 | OpenAI text-embedding-3 | 中英文混合 ESG 场景效果更稳，且零外部成本；OpenAI 嵌入仍可通过开关切换 |
| 向量库 | Pinecone（云）+ 本地缓存 | Milvus / pgvector | 业务量级未到自建必要；后续可平滑迁移 |
| 图谱 | Neo4j | 关系库 + 自实现 | 多跳查询和模式匹配语义化更强，配套生态成熟 |
| 会话存储 | Redis | 数据库 | 多轮上下文要低延迟读取 + 滚动过期，Redis 更合适 |
| 通知队列 | SQLite | Redis / 消息队列 | 当前流量充足、可与主库共享事务边界；接口抽象，量起来再换 |
| 失败处理 | fallback 链 + 闭环上报 | 直接报错 | 优先保证可用性，把"答不出"变成可追踪的数据资产 |

## 6. 可观测性与评测

### 6.1 在线观测

- **分阶段计时**：每次请求返回 `timings_ms`，同时打日志，可在日志系统按字段聚合
- **降级路径监控**：`backend` 字段告诉你这次走到了哪一级
- **路由分布**：按 `retrieval_strategy` 统计每天各策略占比，预警异常分布
- **失败信号率**：`no_context` / `insufficient_context` 占比 < [X]% 为健康

### 6.2 离线评测

`evals/` 目录下：

当前仅检索评测脚本骨架已落地（见 [scripts/rag_retrieval_eval.py](scripts/rag_retrieval_eval.py) 与 Task 3），答案质量与延迟 benchmark 在 Roadmap 中。

| 评测脚本 | 衡量 |
|---|---|
| Retrieval eval | Recall\@k、MRR、空召回率、覆盖率 |
| Answer eval（规划中） | 忠实度、引用准确率、无证据编造率 |
| Latency benchmark | P50 / P95 分阶段延迟、缓存命中率 |

每次有较大改动（路由策略、prompt、模型升级）前后都跑一次评测，结果贴 PR。

## 7. 已知限制 & 后续 Roadmap

### 7.1 已知限制

- 长尾企业（小盘股 / 非英语母语市场）报告覆盖不全
- 表格中跨页合并、图片中嵌入的数字暂未抽取
- 图谱关系抽取在涉及"因果链超过 3 跳"时准确率下降
- 预测模式对"未来 > 3 年"的预测置信度系统性偏低

## 9. 常见问题（FAQ）

**Q: 我问了一个明显在报告里的问题，但答出来说"信息不足"？**
A: 先看返回里的 `retrieval_strategy` 和 `sources`。如果 `sources` 为空，可能是检索没命中（chunk 切分不合适 / 实体没识别到）；如果 `sources` 非空但 LLM 仍说不足，可能是 prompt 过严或证据语义偏离。把 `timings_ms` 和 query 发到 #esg-qa-debug 群里。

**Q: 同一问题不同时候答案不一样？**
A: 默认走 `temperature=0`，差异通常来自：检索结果排序变化（向量库版本更新）、被路由到了不同策略（rewrite 结果不同导致）、走了不同降级级别。返回里的 `backend` 和 `routing` 字段会告诉你走了哪条路径。

**Q: 想限定只查某家公司 / 某年的报告？**
A: 前端勾选筛选条件，后端会把 `document_ids` 或元数据 filter 透传给检索层 + 图谱层。

**Q: 系统答不上来的问题怎么追踪？**
A: 见第 4 节闭环流程；研究员邮箱每 15 分钟收一封 digest，Admin 后台也能实时查看。
